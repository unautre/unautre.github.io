Decorator Pattern in Python

Apr 29, 2016

The decorator pattern is a design pattern introduced in the Gang of Four’s (GoF) book, a classic in software engineering. It is defined as following: there is a base class, and we need to dynamically add behavior. As the saying goes, “classes should be closed to modification but open to extension” ; that is, you shouldn’t have to modify the class everytime you add a new behavior. Thus, behaviors should have their own classes instead of being declared in the base class. That’s what the decorator pattern is all about: a new behavior will take the form of a class, that will have the original object as instance attribute, and serve the same interface as the original object, and change behavior as expected.

class Base(object): # new-style class
	def __init__(self, arg1, arg2):
		self.arg1 = arg1
		self.arg2 = arg2
	def methodA(self):
		return self.arg1 + self.arg2

class Decorator(object):
	def __init__(self, child=None):
		self.bind(child)
	def bind(self, child):
		self.child = child
	def methodA(self):
		return self.child.some_method()*2 # example additional behavior

It works well ! The decorator points back at the original object through the child attribute, and has the same available methods as the Base class.

>>> a = Base(2, 3)
>>> a.methodA()
5
>>> a = Decorator(a)
>>> a.methodA()
10

Now, multiples behaviours can be added, and sometimes stacked. Let’s write a BaseDecorator class and use it to write some more decorators.

class BaseDecorator(object):
	def __init__(self, child=None):
		self.bind(child)
	def bind(self, child):
		self.child = child

class DoubleDec(BaseDecorator):
	def methodA(self):
		return self.child.methodA()*2

class SuccessorDec(BaseDecorator):
	def methodA(self):
		return self.child.methodA()+1

Some usage examples:

>>> a = Base(2, 3)
>>> b = DoubleDec(a)
>>> b.methodA()
10
>>> c = SuccessorDec(b)
>>> c.methodA()
11

Now, what happens if you have another base method, that you need to decorate ? You could just add the method in the decorator, and delegate it to the child object ; but you would then have to add every base method to every decorator. Luckily, Python has some tricks to save us the hassle; namely, the __getattr__ method. The __getattr__ method will be called everytime an attribute is asked to the object, and can’t be found by usual means ; we will use it, so that everytime a method is called on the decorated object, and that method is not overrided in the decorator, it’ll use the base method.

class Base(object):
	def __init__(self, arg1, arg2):
		self.arg1, self.arg2 = arg1, arg2
	def methodA(self):
		return self.arg1 + self.arg2
	def methodB(self):
		return self.arg1 * self.arg2

class BaseDecorator(object):
	def __init__(self, child=None):
		self.bind(child)
	def bind(self, child):
		self.child = child
	def __getattr__(self, name):
		return getattr(self.child, name)

class DoubleDec(BaseDecorator):
	def methodA(self):
		return self.child.methodA()*2

With the preceding implementation for Decorator, a succeful decorator would have to delegate each method of the interface to its decorated object ; a cumbersome task that we leave to the Java devs. Now, a method that is not overridden in the decorator will just use the child object’s method:

>>> a = DoubleDec(Base(3, 4))
>>> a.methodA()
14
>>> a.methodB()
12

Now we can sit back, and enjoy our finely crafted decorator pattern implementation… Or can we ?

>>> a = Base(3, 4)
>>> a.methodA()
7
>>> a.arg2 = 2
>>> a.methodA()
5
>>> b = DoubleDec(Base(3, 4))
>>> b.methodA()
14
>>> b.arg2 = 2
>>> b.methodA()
14

What happened exactly ? When setting the arg2 attribute of b, we set the attribute on the decorator, and this modification didn’t reflect on the decorated object. What do we do ? Well, similarly to the __getattr__ trick of earlier, the __setattr__ method will come to the rescue. This method is called everytime you set an attribute on the object.

class BaseDecorator(object):
	def __init__(self, child=None):
		self.bind(child)
	def bind(self, child):
		self.child = child
	def __getattr__(self, name):
		return getattr(self.child, name)
	def __setattr__(self, name, value):
		if name == "child":
			object.__setattr__(self, name, value)
		else:
			setattr(self.child, name, value)

We have to be careful on what attribute is accessed, and then what we do with it: if we try to set an attribute on self in the __setattr__ method, we will end up in an infinite recursion ; and the child attribute should be looked up directly instead of relying on the child object.

You’ve now been introduced to the decorator pattern in Python ; I said “introduced”, because many improvements are still possible: what happens if the decorator also has attributes ? The function calling syntax is cumbersome, can we improve it ?
